# NL2SQL 视图分析

当查询需要基于 TiDB 视图执行，并且这些视图是通过 `mcporter` 插件暴露出来时，应使用此 skill。

必须通过 `mcporter` 提供的 MCP tools 发起调用，使用其暴露的视图发现、Schema 获取、SQL 执行、表格生成和图表生成能力。

重点要求：

- 必须优先并且仅通过 `mcporter` 对应的 MCP tools 完成相关操作。
- 不要使用 `exec`、shell 命令、手写脚本、直接连库或任何绕过 `mcporter` 的方式。
- 只要 `mcporter` 的 MCP tools 可用，就不得改用其他通道代替。
- 回答中应默认遵循 OpenClaw 原生工具调用链路，让前端能够展示中间工具步骤。

## 工作流

1. 先调用 `list_views`，确认当前可用视图。
2. 如果用户在对话中明确指定了一个或多个视图，应将这些视图视为唯一允许的分析范围，优先级高于任何推断。
3. 针对与问题最相关的视图调用 `get_view_schema`，先看结构再写 SQL。
4. 调用 `run_sql` 获取最终结果集。
5. 使用 SQL 返回的列和行调用 `generate_table`，确保原始结果可以在 UI 中展示。
6. 当问题适合图形化展示时，基于同一份结果继续调用 `generate_chart`。

## 视图范围解析

遇到以下表达时，应将其视为用户显式限定了视图范围：

- `只使用视图：...`
- `仅使用视图：...`
- `仅基于视图：...`
- `只查询视图：...`
- `请不要使用其他视图，只查询：...`
- `请只用 ... 这个视图`
- `请只用以下视图：...`

出现上述表达时，需要遵循以下规则：

- 从用户消息中提取被点名的视图名称。
- 支持用 `、`、`，`、`,`、`；`、`;` 或换行分隔多个视图名。
- 去掉每个视图名前后的空格、标点和引号。
- 即便其他视图看起来也相关，也只能使用用户明确指定的那些视图。
- 如果用户列出了多个视图，则只允许使用该集合。
- 如果某个指定视图在 `list_views` 中不存在，要明确指出，不能静默替换成别的视图。

## 规则

- 如果用户在页面里选择了视图，则只能使用这些视图。
- 如果用户在对话里明确点名视图，应视为与页面选中视图同等效力。
- 优先使用满足问题所需的最小视图集合。
- 写 SQL 之前必须先检查 Schema。
- 使用 TiDB / MySQL 风格 SQL，并对标识符使用反引号。
- 结果集控制在 500 行以内。
- 在最终结果中，必要时使用清晰的中文别名。
- 最终 SQL 执行后，必须生成 `generate_table`。
- 遇到排名、对比、占比、趋势类问题时，应生成 `generate_chart`。
- 优先走 OpenClaw 原生工具调用链路，以便客户端按步骤展示中间过程。
- 如果用户既没有指定视图，也没有在页面中选择视图，则可以结合 `list_views` 与 Schema 检查来推断最合适的视图。
- 使用工具后，应输出专业的中文分析总结，而不是随意的简短回复。
- 所有与数据相关的动作都应默认建立在 `mcporter` MCP tools 的调用基础上，而不是自行替代实现。

## 最终回答风格

最终回答应使用专业中文，默认采用 `汇报型` 风格。

所谓 `汇报型`，指的是：

- 语气正式、清晰，适合汇报场景
- 先讲结论，再补充过程
- 重点突出排名、差距、趋势和对决策有价值的发现
- 结构清爽，便于快速浏览
- 除非用户明确要求，否则不要写得过于技术化

在有数据可用时，最终回答通常应包含以下部分：

1. `分析范围`
   说明使用了哪个或哪些视图，以及指标统计口径。
2. `执行结果`
   简要确认是否生成了 SQL、表格和图表。
3. `核心结论`
   概括主要排名、趋势、对比或聚合发现，必要时附带精确数值。
4. `业务解读`
   用更偏分析和汇报的方式解释结果意味着什么。
5. `说明`
   标注样本范围、仅限指定视图、结果为空、维度不可用等关键信息。

请遵循以下写作规则：

- 默认使用中文。
- 相比口语化表达，要更完整、更专业。
- 优先使用适合项目演示、管理汇报或业务复盘的表达方式。
- 当用户限定了视图范围时，必须明确写出所使用的视图名。
- 关键指标、排名、总量、均值、增长和差距等，应直接引用 SQL 结果中的数据。
- 如果生成了图表，要说明图表类型以及它支撑了什么趋势或对比。
- 如果生成了表格，要提示详细行数据可在表格中查看。
- 不要编造数据中不存在的业务背景。
- 即使结果集很小，也应给出解释，而不只是罗列数字。
- 除非用户明确要求简短，否则不要只说“已完成统计”之类的极简句子。
- 优先使用短段落或带标签的小节，而不是大段堆砌项目符号。
- 在合适时，可用一句简洁的管理视角建议或结论收尾，但必须有数据依据。

当问题是排名或对比类时，优先采用如下结构：

- `分析范围：...`
- `核心结论：...`
- `关键发现：...`
- `图表说明：...`
- `补充说明：...`

当问题是趋势分析类时，优先采用如下结构：

- `分析范围：...`
- `趋势结论：...`
- `阶段特征：...`
- `图表说明：...`
- `补充说明：...`

语气示例：

- 更好：`从院系科研经费分布看，计算机学院处于明显领先位置，科研资源集中度较高。`
- 避免：`我帮你查出来了，第一名是计算机学院。`

## 推荐汇报模板

除非用户明确要求其他风格，否则最终回答默认采用一页式汇报格式，并严格使用以下五个部分：

`标题：`
使用一个简短、偏业务汇报风格的标题，例如 `院系科研总经费排名汇报` 或 `教师年度收入趋势汇报`。

`结果表格：`
在标题下方直接放一个紧凑的 Markdown 表格。表格应包含理解结果所需的关键数据，让读者无需展开工具输出也能看懂结论。

- 排名类问题，展示主要排名对象及其数值。
- 趋势类问题，展示关键时间段及对应指标值。
- 对比类问题，展示被比较对象及其核心指标。
- 优先使用紧凑表格，通常展示前 5 行或最具代表性的几行。
- 如果结果列很多，只保留最有决策价值的列进入最终摘要表格。
- 如果没有有效结果行，写 `无可展示结果`。

`摘要：`
写一个简短段落，通常 2 到 4 句，先概括最重要的结果。它应读起来像可以直接用于演示或汇报的高层摘要。

在 `摘要` 前留一个明显空行，在 `摘要` 后也留一个明显空行，让这一段与上方表格和下方结论形成清晰分隔。

`三条结论：`
当数据支撑时，提供严格 3 条编号结论。优先覆盖：

1. 排名第一或最突出的结果
2. 关键差距、对比关系或结构性差异
3. 整体分布、集中度或趋势层面的含义

`建议：`
提供严格 1 条简短的管理视角建议或结论，但必须建立在返回数据之上。

在 `建议` 前留一个明显空行，使其与上方结论形成清楚分隔。

附加约束：

- 除非用户明确要求，否则不要额外增加其他章节。
- 优先使用完整句子，而不是生硬罗列数字。
- 展示数字时，要附着明确的业务含义。
- 在最终摘要中直接呈现关键返回数据，而不是只做抽象描述。
- 只要有结果行，`结果表格` 下必须渲染 Markdown 表格。
- 各主要部分之间保持一个空行，且 `摘要` 的上下都要额外留出空行。
- `建议` 上方也要额外留空，使其像一个独立结尾。
- 排名类问题，要在摘要或结论中写出主要排名项及其数值。
- 趋势类问题，要在摘要或结论中写出关键时间段和对应指标值。
- 对比类问题，要在摘要或结论中写出被比较对象及其核心指标。
- 如果生成了表格，应把最重要的几行直接体现在最终回答里，而不是只让用户去打开表格看。
- 如果结果不足以支撑 3 条有效结论，仍保留 `三条结论` 这个节名，但条数可以少于 3，不能硬编内容。
- 如果没有生成图表，除非用户要求，否则不要展开讨论图表。
- 如果生成了表格，可以在摘要中顺带说明详细行数据已在表格中提供。
- 如果结果为空，要明确说明没有返回有效数据，避免做牵强解读。
